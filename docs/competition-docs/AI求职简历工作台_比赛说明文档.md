## 模块一：用户需求和问题定义

目标用户为正在准备实习、校招、社招转岗的大学生与青年求职者，也包括需要辅导学生就业的高校就业老师、职业规划顾问与培训机构。对个人用户来说，求职不是只写一份简历，而是一个从个人经历梳理、岗位匹配、简历定制、自荐信撰写、投递跟进到模拟面试复盘的连续流程。传统工具通常只解决其中一个环节，例如简历模板网站只负责排版，文档工具只负责编辑，表格只负责记录投递，模拟面试又需要另找平台，用户很难在同一个工作流中持续积累求职上下文。

当前痛点主要体现在四个方面。第一，经历表达与岗位要求之间存在断层。很多学生有项目、竞赛、实习或课程经历，但不知道如何转换成 HR 和面试官能快速理解的能力证据；同一份简历也常常被重复投递到不同岗位，缺少针对 JD 的结构调整和关键词强化。第二，简历制作与求职材料管理成本高。用户需要在多个文件、多个版本和多个网站之间来回切换，常常出现“最新版在哪里”“这份简历对应哪个岗位”“自荐信用的是哪份简历”等混乱问题。第三，投递与面试准备脱节。投递记录、岗位 JD、简历版本、面试问题和复盘结论通常分散存放，无法形成连续的求职资产。第四，AI 改写结果难以落地。普通聊天式 AI 可以给建议，但无法直接理解简历结构、定位到具体模块，也缺少可确认、可回滚、可导出的编辑闭环。

当用户准备投递一个岗位时，可以使用 AI 求职简历工作台完成从“基础简历创建”到“岗位定制版生成”的全过程。系统支持用户在左侧编辑简历内容，在中间实时预览成品，在右侧呼出 AI Agent 进行润色、诊断、JD 匹配、结构调整与模拟面试。所有 AI 修改都以差异卡片展示，用户确认后才写入简历正文，避免 AI 直接覆盖原有内容。平台还提供自荐信、投递管理、模拟面试、面试报告等功能，使求职材料和求职过程形成统一的数字工作台。

## 模块二：产品功能和流程

AI 求职简历工作台是一个面向 AI-Native 求职流程的在线简历与职业准备平台。它不是简单的简历模板编辑器，也不是只提供一次性回答的聊天助手，而是把简历结构、富文本编辑、实时预览、AI Agent、岗位上下文、投递管理和模拟面试连接成一个可持续使用的工作流。用户可以先建立一份基础简历，再围绕不同岗位生成定制版本，并将每一次投递、沟通、面试和复盘沉淀为后续优化的依据。

产品流程可以概括为：求职管家判断当前阶段 -> 基础简历创建 / 图片导入还原 -> 模块化编辑 -> 实时预览 -> AI 体检、校对、排版、量化 STAR 与润色改写 -> JD 匹配面板与岗位定制版本生成 -> 自荐信生成与导出 -> 投递看板跟进与 AI 阶段建议 -> 公司岗位研究 -> 模拟面试 -> 多轮面试报告复盘 -> 下一轮简历优化。

系统核心功能包括简历用户中心、求职管家 Copilot、简历编辑器、富文本与模块化排版、PDF/图片导出、AI 职业工作区、自荐信管理、投递看板、模拟面试大厅和面试报告大厅。简历编辑器负责个人信息、求职意向、教育背景、项目经历、实习经历、技能证书等模块的结构化编辑；AI Agent 不只负责润色改写，还被拆分为创建助手、图片导入助手、编辑助手、评分诊断、校对纠错、排版美化、量化 STAR、岗位方向推荐、JD 匹配、自荐信生成、模拟面试官和面试分析师等多个角色；投递管理负责记录公司、岗位、渠道、状态、下一步动作和跟进时间；模拟面试负责按岗位背景发起练习或真实模拟，并在多轮面试之间生成面试官交接评价。

### 产品架构图

前端以 Next.js 构建，提供简历中心、编辑器、Career Workspace、投递看板、自荐信、模拟面试与报告页面。服务端通过 API 统一管理简历、自荐信、投递记录、面试记录、Agent 对话状态与 PDF 生成。底层数据主要存储在 SQLite 与本地数据文件中，简历编辑与导出基础来自开源项目 wzdnzd/resume 的 MIT 授权代码，本项目在此基础上进行二次开发，并扩展了 resume-core 相关封装，包括数据规范化、模块排序、富文本转换、AI 可读大纲生成、导出渲染适配和 Agent 操作接口。AI 能力通过 Agent 层调用，围绕简历数据、岗位 JD、投递记录和面试上下文生成可执行建议。

整体架构可以理解为：用户界面层 -> 求职工作区层 -> AI Agent 层 -> 简历核心内核 -> 服务端存储与导出层。

用户界面层负责呈现编辑器、预览、看板、表单和报告；求职工作区层负责把简历、JD、投递、面试场景组织成连续任务；AI Agent 层负责生成修改建议、评分诊断、投递复盘、面试追问和报告；简历基础层在原开源简历编辑能力上扩展数据规范化、AI 大纲、版本关系和导出适配，使简历数据能够稳定转换为可编辑、可预览、可导出和可被 AI 理解的结构；服务端负责持久化数据并通过 Chromium 生成高质量 PDF。

## 操作流程

用户进入平台后，首先在首页点击“开始”，进入我的简历列表。

【图片占位 2：用户中心 / 我的简历列表截图】

首页的求职管家会先读取用户当前的简历数量、最近更新简历、岗位定制版本数量、投递阶段分布、进行中投递和停滞投递等信号，用自然语言判断用户处在“还没开始写简历”“已有简历但未投递”“投递停滞需要跟进”“已有面试需要训练”等阶段，并给出可点击的下一步入口，如创建简历、做 JD 匹配、岗位方向推荐、模拟面试或查看投递。

【图片占位 9：求职管家 Copilot 阶段判断与行动按钮截图】

用户可以创建新简历，也可以复制已有简历作为新版本。系统会自动生成基础模板，包含个人信息、求职意向和常见简历模块。

如果用户已有旧简历截图，可以通过图片导入助手读取图片中的文本和版式信息，生成一份可编辑的简历草稿。对于识别不清或用户指出错误的地方，Agent 会继续通过元素定位和 diff 修改完成修正。

用户在编辑页左侧填写内容，例如姓名、电话、邮箱、教育经历、项目经历、实习经历、技能证书等；中间区域实时显示简历成品预览，方便随时检查版式和内容长度。

【图片占位 3：简历编辑 + 实时预览三栏工作区截图】

用户可以调整模块顺序、模块标题、图标、行列布局、富文本样式与标签信息，使简历既符合岗位需求，也保持清晰排版。

当用户需要 AI 辅助时，打开右侧 Agent 面板。Agent 可以读取当前简历结构，对选中文本进行润色改写，也可以对整份简历进行评分诊断，指出表达空泛、结果缺失、关键词不足、结构顺序不合理等问题。

评分诊断分为两层：本地规则会先检测标题占位、联系方式缺失、空模块、经历缺少数字、占位文本残留、篇幅过长等硬伤；AI 体检再站在 HR 视角生成综合分、五个维度评分、简历亮点和 12 条以内可落地问题。每个问题都带有可直接交给编辑 Agent 执行的 prompt，用户可以从体检报告一键进入修复。

【图片占位 10：AI 体检报告 / 五维评分 / 问题修复入口截图】

当用户准备投递某个岗位时，可以把岗位 JD 输入 Career Workspace。系统会根据 JD 分析岗位关键词与能力要求，建议用户调整项目顺序、强化相关经历、补充技能标签，并可以另存为该岗位的定制简历版本，避免覆盖基础简历。

JD 匹配不是一次性文字回复，而是常驻匹配面板。面板会展示匹配度评分、已命中关键词、缺失关键词、建议列表、建议涉及的目标元素 ID 和一键应用 prompt。用户点击“定位”可以高亮对应简历位置，点击“让 AI 应用”则由 Agent 根据该建议提交待确认修改；简历改动后，系统还能静默重新评分，展示匹配度变化。

【图片占位 11：JD 匹配常驻面板 / 定位与一键应用截图】

用户完成简历后，可以导出 PDF、PNG、JPG、WEBP、SVG 等格式。PDF 默认使用服务端 Chromium 渲染同一份 HTML/CSS，保证预览和导出效果一致；当服务端不可用时，系统会自动降级到浏览器打印。

用户可以基于已保存的简历创建自荐信，填写目标场景、正文内容并导出成 PDF 或文本，用于邮件、内推、官网投递等场景。

用户在投递看板中记录公司、岗位、投递渠道、JD 链接、关联简历、投递阶段、优先级、下一步动作和备注。AI 投递助手可以基于当前阶段给出跟进建议、可复制的 HR 沟通话术和面试准备主题。

【图片占位 6：投递看板截图】

用户可以进入模拟面试大厅，选择简历和岗位背景，发起学习练手或真实模拟。面试开始前，研究 Agent 可以根据公司、岗位或 JD 调用研究服务整理公司业务、岗位能力要求、常见面试重点和来源链接。进入面试后，面试官 Agent 会先内部规划本场问题清单，再逐题展示问题，并根据用户回答进行追问，而不是一次性把题目全部抛出。

真实模拟模式下，系统支持“挂面试”机制。如果候选人连续多题答非所问、核心能力与岗位明显不符或经追问后仍无法给出有效细节，面试官 Agent 可以调用终止工具结束本场模拟，模拟真实招聘中的淘汰压力；学习练手模式则不会启用该终止工具，更适合训练和反馈。

## 竞品差异化

AI 求职简历工作台的创新点不在于单独提供一个“AI 写简历”按钮，而是把求职准备拆解成可管理、可追踪、可复盘的长期流程。传统简历模板工具通常聚焦排版，用户需要自己判断内容是否匹配岗位；普通 AI 聊天工具虽然可以改写句子，但无法稳定理解简历模块结构，也不能把修改结果直接落到编辑器和导出成品里；表格类投递管理工具只能记录状态，不能和简历版本、岗位 JD、面试准备形成联动。

与现有方案相比，本项目主要有以下差异。

第一，从“模板填空”变为“结构化求职资产”。平台把简历拆成个人信息、求职意向、经历模块、标签、富文本片段等可操作结构，后续 JD 匹配、AI 修改、PDF 导出和面试准备都基于同一份结构化数据，而不是散落的文本。

第二，从“一份简历投所有岗位”变为“基础简历 + 岗位定制版本”。用户可以保留一份母简历，同时针对不同岗位另存定制版。系统记录定制版与母简历之间的关系，方便用户理解每个版本服务于哪个岗位。

第三，从“AI 给建议”变为“AI 可确认地改文档”。Agent 不直接覆盖用户内容，而是以 diff 卡片呈现修改建议。用户确认后才应用到简历中，这让 AI 辅助更适合严肃求职场景。

第四，从“简历制作”扩展为“求职全流程工作台”。平台同时覆盖自荐信、投递看板、模拟面试、面试交接和复盘报告，使用户在一个系统内完成从准备材料到拿到反馈的闭环。

第五，从“单次问答”变为“持续上下文”。投递记录、面试记录、Agent 对话、简历大纲和岗位信息会被保存，后续 AI 可以基于已有上下文继续给出更贴近用户实际情况的建议。

第六，从“一个万能聊天助手”变为“多角色 Agent 协作”。平台把创建、图片导入、编辑、校对、排版、量化 STAR、评分诊断、JD 匹配、岗位方向推荐、自荐信、面试官、面试分析和报告生成拆成不同模式，每个模式有独立 prompt 与工具权限。这样既能提升场景专业度，也能降低 AI 在错误场景中乱改内容的风险。

第七，从“面试题库”变为“研究 + 规划 + 追问 + 报告”。普通面试工具往往只是给一组题，本项目会先研究公司与岗位，再根据简历和轮次规划问题，逐题追问，并把多轮记录生成结构化复盘报告，包含六项能力雷达和逐题参考答案。

## 模块三：AI 原理和技术说明

### AI 功能说明

求职管家能力：系统首页内置 Copilot，会读取简历列表、投递记录、岗位定制版本和阶段分布，判断用户当前最需要做什么。它不是单纯聊天，而是会返回可点击行动按钮，把用户带到创建简历、AI 润色、JD 匹配、岗位方向推荐、模拟面试或投递看板等具体工具。

简历结构解析能力：系统将简历转换为 AI 可理解的大纲，包含个人信息、求职意向、模块标题、经历内容、标签、行列布局、元素 ID、富文本纯文本摘要和样式信息。AI 不需要从页面截图中猜测内容，而是直接读取结构化简历数据，并能定位到具体模块、行和元素。

对话式创建能力：创建助手可以从空白简历开始，通过一问一答收集基本信息、求职意向、教育经历、项目经历、工作经历和技能，并调用结构化工具写入简历。它会主动采用教育背景四列、工作经历标题行四列、项目经验标题行三列、详情行一列要点等规范，避免把所有内容堆进一个文本框。

图片导入能力：用户可以上传旧简历截图或照片，图片导入助手会读取图片内容并调用 replace_resume 生成完整简历草稿。后续用户指出识别错误时，Agent 会先定位对应模块和元素，再提交待确认修改。

润色改写能力：Agent 可以针对用户选中的某段经历进行表达优化，将“做了什么”改写为“在什么背景下，用什么方法，产生什么结果”，提升简历中的动作动词、量化结果和岗位关键词密度。量化 STAR Agent 会特别关注金额、百分比、用户规模、效率提升、排名等结果指标；如果原简历没有数字，系统要求 Agent 追问用户，而不是编造。

校对与排版能力：校对 Agent 只处理错别字、语病、标点、日期格式、术语大小写和空格等客观错误，不做风格改写；排版 Agent 只调整主题色、个人信息布局、标题居中、头像形状、模块顺序、字号、加粗和对齐，不改正文事实。不同 Agent 挂载不同工具集，降低越权修改风险。

结构诊断与 AI 体检能力：系统可以根据简历模块顺序、内容完整度、经历相关性和表达清晰度给出评分与修改建议，帮助用户发现空泛表达、重复信息、重点不突出等问题。AI 体检会输出综合分、内容完整性、量化成果、岗位匹配、表达清晰、排版样式等维度评分，并把每个问题转化为可一键交给编辑 Agent 的修复 prompt。

JD 匹配能力：用户输入岗位 JD 后，Agent 会分析岗位职责、任职要求和关键词，再对照简历大纲指出匹配项、缺口项和可强化内容。必要时可以生成岗位定制简历版本。

岗位方向推荐能力：当用户不知道投什么岗位时，岗位方向推荐 Agent 会基于专业、技能、项目和经历反推出 3-5 个适合方向，并给出匹配度、推荐理由、典型岗位和当前能力缺口。该模式只读简历，不挂载修改工具，避免未经用户同意更改内容。

投递复盘能力：投递看板中的公司、岗位、阶段、渠道、简历版本、跟进时间等信息会被汇总给 AI，AI 根据阶段分布、停滞记录、渠道表现和优先级给出投递策略建议。

自荐信生成能力：自荐信 Agent 只能读取简历并写入自荐信，不允许修改简历。它会根据正式求职信、简短开场白、内推说明或通用版本等场景生成草稿，并保留引用的简历依据和短版话术。

公司与岗位研究能力：模拟面试前，研究 Agent 可以根据公司、岗位或 JD 调用研究服务，整理公司业务、近期动态、岗位能力要求、技术栈、常见面试重点和来源 URL，为后续出题提供依据。

模拟面试能力：系统基于简历内容和岗位背景生成面试问题，在学习练手模式下帮助用户逐步练习，在真实模拟模式下根据回答质量推进或终止面试。面试官 Agent 会先内部规划 3-5 道题，包含题型、难度、考察维度和追问方向，再逐题展示；多轮面试之间可以生成上一轮面试官交接评价，让后续追问更接近真实企业流程。

面试报告能力：系统根据一轮或多轮面试问答记录生成复盘报告，包括综合分、综合评级、六项能力雷达、分轮评价、逐题星级、用户原回答要点、评价、参考答案、改进建议和推荐学习资源。

### AI 如何解决痛点

首先，AI 结构解析解决了“简历与岗位断层”的问题。系统不是只让 AI 看一段散文，而是把简历拆成可理解的模块和经历，使 AI 能够判断哪些经历更适合某个岗位，哪些内容需要前置、强化或删减。

其次，AI 改写与 diff 确认解决了“建议难落地”的问题。用户不需要把 AI 输出复制到文档里反复粘贴，系统可以把修改建议直接绑定到简历数据结构上，并通过确认机制降低误改风险。

再次，JD 匹配解决了“同一份简历重复投递”的问题。平台可以围绕具体岗位创建定制版本，让用户在不破坏基础简历的前提下进行岗位化表达。

然后，投递复盘解决了“投了很多但不知道问题在哪”的问题。AI 可以从投递阶段分布、等待时间、渠道与岗位类型中发现卡点，并给出下一步动作。

最后，模拟面试和报告解决了“简历写完但不会讲”的问题。平台会把简历中的经历转化为面试追问，再把用户回答转化为复盘建议，让简历优化和面试训练互相反哺。

如果去掉 AI，平台仍然可以作为普通简历编辑器、导出工具、投递记录表和面试记录管理器使用，但无法自动完成 JD 匹配、经历改写、投递复盘和面试追问，也无法把用户的求职数据转化为可执行建议。因此 AI 是本项目从“简历工具”升级为“智能求职工作台”的关键。

### AI 技术路线

本项目采用“结构化简历数据 + 多 Agent Profile + 工具调用 + ChangeSet 确认 + 服务端 API”的技术路线。前端使用 Next.js 构建页面和交互，编辑器负责采集用户输入和展示预览；简历数据结构与基础编辑能力基于 wzdnzd/resume 开源项目二次开发，本项目进一步补充了 ResumeData 版本关系、富文本规范化、AI 可读大纲、Agent 操作接口和求职场景数据联动；Agent 层接收简历大纲、选区内容、JD、投递记录或面试上下文，生成结构化建议；服务端 API 负责数据持久化、AI 请求转发、PDF 生成和状态管理。

多 Agent Profile 是系统 AI 能力拆分的核心。不同场景使用不同 system prompt 和工具白名单，例如编辑助手可以使用 update_element_text、add_rows、reorder_modules 等编辑工具；评分诊断只允许读取简历并展示评分卡片；校对纠错只允许文本修正；排版美化只允许布局、主题色和样式调整；岗位方向推荐只读简历；自荐信只能生成信件草稿；真实模拟面试才挂载 terminate_interview 工具。通过这种方式，系统把“AI 能做什么”和“AI 不能做什么”在工具层做了边界约束。

在简历优化阶段，系统先将 ResumeData 转换为大纲文本和模块索引，再把用户选区、目标任务和岗位 JD 传给 Agent。Agent 返回修改建议、诊断卡片或结构化操作，前端以卡片形式展示，用户确认后调用简历操作接口更新对应字段。

所有修改类工具都不是直接改数据，而是先生成 ChangeSet。ChangeSet 中记录操作类型、摘要、目标 ID、修改前内容、修改后内容、补充说明和 apply 纯函数。前端把 ChangeSet 渲染成 diff 卡片，用户可以接受或拒绝；被接受的修改才真正写入简历，被拒绝的修改会写入上下文，提醒 Agent 不要重复提交同样方案。这是本项目区别于普通 AI 对话复制粘贴的关键机制。

在 AI 体检阶段，系统先执行本地规则预检，检测联系方式、空模块、占位文本、量化不足、篇幅偏长等确定性问题，再把预检结果和简历大纲交给模型生成 JSON 结构化报告。报告必须包含 overallScore、五个维度评分、真实亮点和问题列表；每条 issue 都包含 evidence、suggestion 和 prompt，形成“诊断 -> 一键修复”的闭环。

在 JD 定制阶段，系统读取岗位描述，提取岗位职责、能力要求、关键词和优先级，再与简历大纲进行比对。AI 输出包括匹配度判断、已命中关键词、缺失关键词、经历排序建议和可替换文案。每条建议会带 targetIds 和 prompt，targetIds 对应 get_resume 大纲中的 element、row 或 module，方便界面定位和高亮；用户可以将优化结果另存为岗位定制版，形成母简历与子简历的版本关系。

在投递管理阶段，系统把投递记录压缩为结构化摘要，包括公司、岗位、状态、投递天数、下一步动作、关联简历和 JD 摘要。AI 根据这些数据生成复盘报告、跟进话术和面试准备主题。

在投递建议阶段，系统不只看单条岗位，还会汇总全部投递数据，统计阶段分布、进行中投递、停滞时间、关联简历情况和下一步动作。AI 输出 observations、risks、nextActions 等结构化建议；单条投递助手还会根据当前阶段生成 followUpMessage 和 interviewTopics，使“投递后该做什么”具体化。

在模拟面试阶段，系统把简历大纲、岗位背景、轮次设定和历史问答作为上下文。面试官 Agent 负责计划题目和逐题追问，分析 Agent 负责单题评分与回答优化，报告 Agent 负责生成最终展示结果。多轮面试之间通过 handoff 机制把上一轮评价传递给下一轮，使模拟过程更接近真实招聘流程。

面试报告生成采用强制工具输出。报告 Agent 必须调用 present_campaign_report，返回 overallScore、overallLabel、summary、competencies、rounds 和 suggestions。competencies 固定为基础知识、项目经验、工具使用、逻辑思维、沟通表达、业务理解六项；rounds 只覆盖用户实际选定的轮次；每道题保留 question、answer、evaluation、referenceAnswer 和 starRating，保证报告可以稳定渲染成能力雷达、分轮评价和逐题复盘。

整体工作流可以概括为：简历/岗位/投递/面试数据 -> 结构化摘要 -> Agent 任务生成 -> 工具调用 -> ChangeSet 或展示卡片 -> 用户确认 / 面板交互 -> 数据写回 -> 导出、投递、面试或复盘。

## 模块四：功能实现和商业化

### 功能完成度

当前项目已经具备较强的可运行性和扩展性。前端已实现欢迎页、简历中心、简历编辑器、实时预览、导出菜单、自荐信管理、投递管理、模拟面试大厅、Career Workspace、面试报告页面等主要界面。后端已实现简历、自荐信、投递、面试记录、面试报告、Agent 状态和 PDF 生成等 API。数据层通过 SQLite 与本地文件保存运行时数据，并兼容旧数据迁移。

简历编辑方面，系统支持个人信息、求职意向、模块化经历、标签、富文本、布局调整、头像、预览和导出。导出方面，支持服务端 Chromium 生成 PDF，也支持图片格式导出和浏览器打印降级。AI 工作区方面，已形成围绕简历、JD、投递和面试的 Agent 面板，支持对话、选区、工具调用、ChangeSet 待确认修改、历史 diff 状态、评分卡片、JD 匹配卡片、岗位方向卡片、面试题卡片和面试报告卡片。

AI 深度功能方面，当前已具备求职管家 Copilot、图片导入简历还原、对话式简历创建、AI 简历体检、本地规则预检、校对纠错、排版美化、量化 STAR 改写、JD 匹配常驻面板、匹配度重评、岗位方向推荐、自荐信生成、投递数据复盘、单条投递阶段助手、公司岗位研究、模拟面试内部题目规划、逐题追问、真实模拟终止机制、多轮面试交接评价和综合面试报告生成等能力。这些能力不是简单地把用户输入发送给模型，而是围绕结构化简历和求职流程设计了专门的工具 schema、结果卡片和确认机制。

当前仍有进一步优化空间。第一，AI 服务提供商和模型配置可以继续扩展，让用户自由选择 OpenAI、Anthropic、Gemini 或国产模型等不同接口。第二，简历模板样式还可以增加更多行业风格。第三，远程同步和多端协作可以通过 WebDAV、OneDrive、Google Drive 等方式增强。第四，面试题库、岗位知识库和行业数据还可以继续补充，使 AI 建议更专业。

主要开源依赖包括：Next.js、React、React DOM、TypeScript、Tailwind CSS、PostCSS、ESLint 等前端工程与构建基础；Radix UI、shadcn/ui 风格组件、class-variance-authority、clsx、tailwind-merge、tailwindcss-animate、tw-animate-css 等界面组件与样式辅助；Tiptap、ProseMirror 相关包用于富文本编辑；Iconify、lucide-react 用于图标展示；@hello-pangea/dnd 用于拖拽排序交互；puppeteer-core 与 @sparticuz/chromium 用于服务端 PDF 渲染；html-to-image 用于图片导出；next-themes、date-fns、geist 等用于主题、日期和字体体验。

### 商业化思路

AI 求职简历工作台的商业价值不只是帮助用户做一份更漂亮的简历，而是帮助用户提高求职准备效率、岗位匹配质量和面试表现。它可以面向 C 端求职者、高校就业中心、职业培训机构和企业校招服务场景拓展。

盈利模式可以包括以下几类。

个人订阅收入：向个人用户提供免费基础简历编辑和导出，高级功能如 JD 深度匹配、多版本定制、AI 模拟面试、完整面试报告和批量投递复盘可采用月度或年度订阅。

按次 AI 服务收入：对于高消耗功能，例如深度简历诊断、岗位定制版生成、真实模拟面试、多轮面试报告等，可以按次计费或使用积分包。

高校与机构 SaaS 收入：为高校就业中心、学院、训练营和职业培训机构提供批量账号、学生简历诊断、就业数据看板、模拟面试训练和简历作品集管理。

企业招聘辅助服务：与企业校招或实习项目合作，为候选人提供标准化简历提交、岗位匹配、面试准备和材料预审工具，提高招聘前期沟通效率。

增值服务收入：围绕职业咨询、简历精修、模拟面试陪练、行业岗位包、模板市场等建立增值服务。

## 总结

AI 求职简历工作台把传统分散的求职准备流程整合为一个连续的智能工作台。它用结构化简历数据保证内容可编辑、可预览、可导出，用多 Agent 与工具调用解决创建、导入、润色、校对、排版、量化、诊断、JD 匹配、投递复盘、公司研究、模拟面试和报告生成问题，用 ChangeSet 确认机制保证 AI 修改可审阅、可拒绝、可追踪，用投递与面试记录把求职过程沉淀为可复用资产。相比普通简历模板工具，本项目更强调从“写简历”走向“经营求职过程”；相比普通 AI 对话工具，本项目更强调结构化数据、工具权限、可确认修改和真实落地。

项目已经完成从简历创建、AI 优化、材料导出、投递管理到面试复盘的核心闭环，具备继续扩展为面向个人、高校和培训机构的 AI 求职服务平台的潜力。
